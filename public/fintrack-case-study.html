Project overview

The context

70% of people under 35 say they struggle to save money, yet most budgeting apps have steep learning curves, cluttered dashboards, and require 10+ minutes of setup before delivering any value. Users download finance apps, get overwhelmed, and delete them within a week. FinTrack was designed from scratch to break that cycle.

05 — Design Solutions

Key design decisions

⚡
Zero-setup dashboard
The app provides value from the very first launch using smart category defaults and estimated patterns. No bank connection or configuration required to start seeing useful data.
💬
Persistent one-tap logging
A floating action button is accessible from every screen in the app. The numpad opens automatically, reducing the logging flow to 3 taps and under 20 seconds.
🎯
Positive reinforcement system
A daily streak for consistent logging replaces guilt-based overspending alerts. Users earn visual rewards for building the habit — not punished for breaking it.
📈
Single key metric dashboard
The dashboard shows one number prominently: "Remaining this week." All other data is secondary and collapsed. This directly reduces cognitive overload from competing data points.
06 — Testing & Iteration

What broke and
how I fixed it

I ran two rounds of unmoderated usability tests with 5 participants each using Maze. Participants completed 3 core tasks: log an expense, set a budget goal, and find last month's total spending.

✕ Round 1 — Issue found
60% of users could not find the expense history. It was buried in a bottom-navigation tab labelled "Records" — a word that meant nothing to most users.
✓ Iteration applied
Renamed to "History" and added a swipe-down gesture from the home dashboard. Task completion jumped from 40% to 95% in round two.
✕ Round 1 — Issue found
Users completely skipped the budget goal feature. They didn't understand what "Add envelope" meant — UX jargon borrowed from envelope budgeting methodology.
✓ Iteration applied
Renamed to "Set a saving goal" with a plain illustration. First-time goal creation improved by 3× in the second round of testing.

08 — Key Learnings

What this project taught me

01
Radical simplicity is a design decision, not a shortcut
The biggest UX problem in finance apps is not missing features — it is too many features shown at once. Choosing to show one number on the dashboard instead of twelve was the hardest and most impactful decision in the entire project.
02
Naming matters more than visuals
Renaming one navigation tab fixed a task that weeks of visual redesign had failed to solve. Always test your labels with real users — what makes sense to a designer rarely makes sense to everyone else.
03
Emotional framing of data is a legitimate design tool
The same spending number presented as "you overspent by 2,000" versus "you have 14,200 left this week" produces completely different user behavior. Framing is design. How you present truth is as important as the truth itself.
